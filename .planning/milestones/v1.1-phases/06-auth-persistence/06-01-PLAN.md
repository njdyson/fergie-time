  <files>server/routes/auth.ts, server/routes/auth.test.ts, server/types.d.ts</files>
  <behavior>
    - register: POST /api/auth/register with {teamName, password} creates save row, hashes password with bcrypt (10 rounds), sets session.saveId and session.teamName, returns {success: true, teamName}
    - register duplicate: POST /api/auth/register with existing team name returns 409 {error: "Team name already taken"}
    - register validation: POST /api/auth/register with empty teamName or short password returns 400
    - login: POST /api/auth/login with correct credentials sets session and returns {success: true, teamName}
    - login wrong password: POST /api/auth/login with wrong password returns 401 {error: "Invalid credentials"}
    - login unknown team: POST /api/auth/login with nonexistent team returns 401 {error: "Invalid credentials"}
    - hash verification: After register, the password_hash in the DB is NOT the plain text password (verify with bcrypt.compare)
  </behavior>
  <action>
    1. Create `server/types.d.ts` with cookie-session module augmentation adding `saveId?: number` and `teamName?: string` to CookieSessionObject.

    2. Write RED tests in `server/routes/auth.test.ts`:
       - Tests need a fresh in-memory or temp-file SQLite database for isolation. Use the real `getDb()` but point it at a temp path, OR create a test helper that initializes a fresh DB. Since `getDb()` is a singleton with a hardcoded path, the simplest approach is to import `Database` from better-sqlite3 directly in tests, create an in-memory DB, run the schema creation SQL, and mock/inject it. However, the routes call `getDb()` internally.
       - Recommended test approach: Use supertest against the Express app. Import `app` from `server/index.ts`. This will use the real DB file in `data/`. Clean up test rows after each test. OR, better: create a test-specific app setup that uses a temp DB.
       - Simplest viable approach: Import the app, use supertest, and clean up rows by team_name after each test. The dev DB is disposable.
       - Test cases from <behavior> block above.
       - Run tests — they MUST fail (routes don't exist yet).

    3. Write GREEN implementation in `server/routes/auth.ts`:
       - Import Router from express, bcrypt from bcryptjs, getDb from ../db.js, z from zod
       - Define AuthBody schema: z.object({ teamName: z.string().min(1).max(50), password: z.string().min(4).max(100) })
       - POST /api/auth/register: validate body, check uniqueness, hash with bcrypt.hash(password, 10), INSERT into saves with game_state='{}' and version=1, set session.saveId and session.teamName, return {success: true, teamName}
       - POST /api/auth/login: validate body, SELECT id,password_hash from saves WHERE team_name=?, bcrypt.compare, set session, return {success: true, teamName}
       - Export authRouter

    4. Run tests — they MUST pass.

    Install bcryptjs first: `npm install bcryptjs && npm install -D @types/bcryptjs`
